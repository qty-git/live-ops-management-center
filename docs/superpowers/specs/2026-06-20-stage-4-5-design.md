# 阶段 4 + 阶段 5 设计说明

## 目标与边界

本轮只实现时间轴中的本人任务高亮、任务相关编辑边界，以及本地账号与权限管理。所有登录用户继续看到全员全天计划；不实现成员视角切换，不扩展为全系统阶段 7 操作鉴权，不修改 CloudBase 时间轴同步协议。

## 方案选择

考虑过三种方案：

1. 为任务新增 `assignedUserId` 并迁移全部时间轴数据。匹配最直接，但会扩大业务数据和云端同步变更面，本轮不采用。
2. 仅按任务负责人姓名匹配。改动小，但重名、改名和岗位兜底不足，本轮不采用。
3. 兼容扩展 users，并通过 `personId → TeamPerson` 解析绑定人员，同时保留姓名、岗位、用户 ID 兜底。无需改时间轴存储结构，符合现有数据约束，本轮采用。

## 权限模型

继续使用字符串权限快照，并补齐阶段 5 页面所需的细粒度权限。需求名通过统一映射转换为内部权限键，避免页面中散落角色判断。

- `timeline:view`：查看时间轴及全员全天计划。
- `task:highlight_own`：显示本人任务高亮。
- `task:edit_own` / `task_feedback:edit_own`：编辑本人状态、问题、方案、备注。
- `task:edit_all` / `task_feedback:edit_all`：编辑全部任务反馈。
- `timeline:manage`：修改任务结构、时间、分配、事件和次日计划。
- `users:manage`：查看和维护账号资料、状态、密码。
- `permissions:manage`：查看并修改权限复选框。

角色只提供新账号的默认权限；每个账号仍保存独立权限快照。普通 admin 默认可管理账号但不能分配权限；super_admin 拥有全部权限。普通 admin 对 super_admin 的编辑、禁用、重置密码和角色修改全部在 userStore 写入口拒绝。

## users 与 session 兼容

users 从 v2 升级为 v3，新增岗位字段并补齐新增权限；保留原账号、密码哈希、人员绑定和时间字段。有效仓库缺失系统超级管理员时补回默认记录，不清空其他用户。

session 从 v3 升级为 v4，并在应用启动及 users 变更后从 userStore 重新解析真实用户。账号已禁用或被删除时清除会话；权限或资料变化时刷新当前会话。密码继续使用现有 SHA-256 浏览器哈希方案，不保存明文。

## 时间轴行为

`TimelinePage` 接收当前真实用户并集中计算能力：是否为本人任务、能否编辑本人反馈、能否编辑全部反馈、能否管理时间轴。`TimelineBoard` 只负责展示高亮和触发详情；`TaskEditor` 根据字段能力决定可编辑/只读，并显示明确权限提示。保存入口再次比较原任务与新任务，只允许相应字段通过，防止仅靠 disabled 控件。

本人任务按以下顺序匹配：绑定人员姓名与 `personName`；账号岗位与任务 `role`；未来存在的 `assignedUserId` 与用户 ID。其他任务不隐藏。普通成员可打开所有详情，但只能改本人 `status`、`issues/problem/solution`、`note`；结构字段、时间、负责人和删除不可用。具有全部反馈或时间轴管理权限的账号按权限放开。

没有 `timeline:manage` 的账号不能创建、删除或编辑事件、次日计划、模板覆盖和时间轴结构；日期切换、查看范围拖动缩放保留。导出继续按 `exports:use` 判断。

## 账号与权限页面

`#/users` 入口与守卫接受 `users:manage` 或 `permissions:manage`。页面包含用户表格、新增/编辑资料、权限弹窗、重置密码和启停操作。岗位与绑定人员从现有 TeamPerson 列表选择，管理岗位允许不绑定排班人员。

权限按需求分组展示。没有 `permissions:manage` 的管理员不渲染权限复选框和权限编辑操作。只有 super_admin 可创建或修改 super_admin；系统超级管理员不能被普通 admin 变更、禁用或重置。

## 错误处理与验证

表单校验账号唯一、必填字段、密码非空和角色边界；userStore 返回明确错误供页面展示。高风险操作使用确认提示。验证覆盖 lint、build，以及 admin/member 的菜单、直达守卫、全员任务可见、本人高亮、字段编辑边界、账号新增、禁用、重置密码和权限编辑。
